# ADR-0004: `path_regex`を`path`とは別プロパティとして導入する

* ステータス: 承認済み

## 決定

パスの完全一致を表す`path`は引き続き文字列プロパティとして維持し、パス全体に対する正規表現条件は別プロパティの`path_regex`として表現します。

`path`自体を文字列とオブジェクトのユニオンにはしません。

## 理由

現状の`path`は、スキーマルートからの具体的な相対パスを指す単純なプロパティとして機能しています。
この役割は、`require`や`forbid`のような単一対象を扱うルールにおいて分かりやすく、実装も単純です。

ここに正規表現指定を同居させると、同じ`path`という名前で

* 単一の具体パス
* 複数候補を選択する条件

の2つの意味を持つことになります。

これは、仕様の理解と実装の両方を複雑にします。

また、本プロジェクトのドラフトでは正規表現系の命名を`*_regex`に揃える方針を採っています。
そのため、パスに対する正規表現条件も`path_regex`とする方が一貫しています。

## 影響

スキーマの利用者は、完全一致と正規表現を明示的に書き分けられます。

* 具体的な1パスを指定する場合は`path`
* ルート相対パス全体に対する正規表現条件を指定する場合は`path_regex`

この方針により、既存の`path`利用との後方互換を維持できます。

一方で、`path_regex`の評価対象、完全一致か部分一致か、`case_sensitive`との関係などは別途仕様として定義する必要があります。
