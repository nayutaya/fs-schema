# ADR-0006: `implies`ルールを追加する

* ステータス: 承認済み

## 決定

FS Schemaに、単一パス間の依存関係を表す`implies`ルールを追加します。

初期仕様の`implies`は、次の形に限定します。

```yaml
- type: implies
  if:
    path: secrets.enc
    exists: true
  then:
    path: secrets.key
    exists: true
```

このとき、`if`側のパスが存在するなら`then`側のパスも存在しなければなりません。

また、初期仕様では次の制約を設けます。

* `if`と`then`では`path`のみを許可する
* `if.path_regex`および`then.path_regex`は許可しない
* `exists`は`true`のみを許可する
* `if`と`then`は文字列ではなくオブジェクトとして表現する

## 理由

`require`と`forbid`に続く次のルールとしては、単一パスどうしの依存関係を表せる`implies`が最も自然です。

たとえば「`config.local.yaml`が存在するなら`config.yaml`も必要」「`secrets.enc`があるなら`secrets.key`も必要」といった条件は、実務上よく現れます。
これらは存在必須や存在禁止だけでは表現しにくく、専用の依存ルールとして持つ価値があります。

一方で、`implies`を最初から一般化しすぎると、仕様と実装の複雑さが急に増えます。
そのため、初期仕様では「単一パスの存在が、別の単一パスの存在を要求する」という最小形に限定します。

## 検討したが採用しなかった案

### `path_regex`を許可する案

`if.path_regex`や`then.path_regex`を許可すると、`implies`は単一対象の依存ではなく、集合に対する依存ルールになります。
すると、少なくとも次の論点を追加で決める必要があります。

* `if.path_regex`が複数件に一致した場合、`then`は各一致ごとに評価するのか
* `then.path_regex`は1件でも一致すればよいのか、すべての一致対象を評価するのか
* `if`側と`then`側の一致結果をどのように対応付けるのか
* 一致件数が0件の場合を成功とみなすのか

これらは、単純な依存ルールというより集合選択ルールの設計に属する問題です。
本プロジェクトでは、`path_regex`はパス集合の選択に使う性質が強いため、この種の要件は将来の`for_each`や`count`など、集合を扱う別ルールとして設計する方が適しています。

### `exists: false`も許可する案

`if.exists: false`や`then.exists: false`を許可すれば、条件付き禁止や否定条件も表現できます。
しかしその場合、`implies`は単純な存在依存ではなく、小さな条件言語に近いものになります。

すると次に、`kind`、`size`、`name_regex`なども条件として載せたくなり、ルールの責務が広がりすぎます。
初期仕様としては、`exists: true`だけを許可し、単純な存在依存に留める方が妥当です。

### `if`と`then`を文字列に簡略化する案

`exists: true`しか許可しないのであれば、`if`と`then`を単なるパス文字列として表す案も考えられます。
たとえば、`if_path` / `then_path`のような専用プロパティにすれば、現状の意味だけをより短く表現できます。

しかし今回は、この簡略化は採用しません。
将来的に`if`側や`then`側へ条件を追加したくなった場合、文字列形式からオブジェクト形式へ再設計する必要があり、スキーマ形式の変更が大きくなるためです。

そのため、現時点では`exists: true`が冗長であっても、`if`と`then`はオブジェクトとして維持します。

## 影響

`implies`により、単一パス間の依存関係を素直に記述できるようになります。
`require`や`forbid`だけでは表現できなかった条件付き依存を、既存のルール体系に近い形で追加できます。

一方で、`implies`は意図的に限定された初期仕様となります。
集合を対象にした依存、否定条件を含む依存、複数条件の組み合わせなどは、このルールでは扱いません。

この方針により、`implies`は初期段階でも実装しやすく、利用者にとっても意味が読み取りやすいルールになります。
