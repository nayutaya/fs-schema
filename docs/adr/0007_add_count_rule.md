# ADR-0007: `count`ルールを追加する

* ステータス: 承認済み

## 決定

FS Schemaに、対象集合の一致件数を検証する`count`ルールを追加します。

初期仕様の`count`は、次の形に限定します。

```yaml
- type: count
  select:
    path_regex: '^logs/[0-9]{8}\.log$'
  minimum: 1
  maximum: 10
```

このルールは、`select.path_regex`に一致するエントリ数が`minimum`以上かつ`maximum`以下であることを要求します。

また、初期仕様では次の制約を設けます。

* `select`では`path_regex`のみを許可する
* 件数の下限には`minimum`を使う
* 件数の上限には`maximum`を使う
* `min`および`max`は採用しない

## 理由

`count`は、「1つ以上存在する」「1つも存在しない」「ちょうど1つ存在する」といった件数条件を明示的に表現できるルールです。
これは、単一パスを扱う`require`や`forbid`、依存関係を表す`implies`では直接表現しにくい要件を扱うために有用です。

一方で、`count`を最初から一般化しすぎると、集合選択の設計が大きくなります。
`under`、`recursive`、`kind`、`name_regex`、`paths`などを同時に導入すると、`count`ルール自体の追加よりも`select` DSLの設計が中心課題になってしまいます。

そのため、初期仕様では`select.path_regex`のみに限定します。
これにより、既存の`path_regex`の意味と整合したまま、集合選択を最小限の追加で表現できます。

また、件数の境界を表すプロパティ名には`minimum`と`maximum`を採用します。
`min`や`max`でも意味は通じますが、短縮形は文脈依存で読み取りづらく、将来プロパティが増えたときに意図がやや曖昧になります。
`minimum`と`maximum`であれば、件数境界を表すプロパティであることが明確です。

## 検討したが採用しなかった案

### `select`で複数の選択条件を許可する案

`select.under`、`select.kind`、`select.name_regex`などを同時に導入すれば、より表現力の高い集合条件を早い段階で扱えます。

しかしこの案では、各条件の組み合わせ方、評価順序、空集合時の扱いなどを一度に決める必要があり、初期導入としては重すぎます。
今回は、まず件数ルール自体を導入することを優先し、対象選択は`path_regex`だけに限定します。

### `min` / `max`を採用する案

短い名前の方が記述量は少なくなります。
しかし、`count`が件数境界を表すルールであることを明確にするには、`minimum` / `maximum`の方が読みやすく、仕様としても説明しやすいです。

また、他の制約ルールで境界値を扱う場合にも、短縮形よりフルスペルの方が一貫した命名方針を作りやすくなります。

## 影響

`count`により、対象集合の件数条件をスキーマで直接表現できるようになります。
これにより、「少なくとも1件必要」「1件も存在してはいけない」「上限件数を超えてはいけない」といった要件を、既存ルールに頼らず明示的に記述できます。

一方で、初期仕様の`count`は意図的に限定されたものになります。
対象集合の指定は`path_regex`のみであり、より複雑な選択条件は今後の拡張課題として残ります。

この方針により、`count`は小さく導入しつつ、後から`select`を拡張できる構造を保てます。
