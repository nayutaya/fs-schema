# ADR-0009: ルールをbashスクリプトに変換する`compile`コマンドを追加する

* ステータス: 承認済み

## 決定

FS Schemaに、スキーマからスタンドアロンなbashスクリプトを生成する`compile`コマンドを追加します。

```sh
fs-schema compile --schema schema.yaml --output check.sh
fs-schema compile -s schema.yaml -o -
```

生成されたスクリプトは、第1引数で指定されたディレクトリ（省略時は`.`）に対してスキーマを検証し、診断メッセージを標準エラー出力に`[code] message`形式で出力します。
すべてのルールに適合した場合は終了コード0、適合しないルールがあった場合は終了コード1で終了します。
ルートが存在しない場合は終了コード2で終了します。

`compile`コマンドは、初期仕様で次のルールに対応します。

* `require`（`path`および`path_regex`）
* `forbid`（`path`および`path_regex`）
* `implies`
* `count`

`options.case_sensitive`および`options.follow_symlinks`にも対応します。

## 理由

`fs-schema validate`はPythonランタイムを必要とするため、Pythonをインストールできない環境（軽量なCIランナー、Dockerイメージ、組み込みシェル環境など）では利用しづらい場面があります。
スキーマからbashスクリプトを生成できれば、Pythonに依存せず`bash`と`find`があれば検証を実行できるようになります。

また、生成されるスクリプトは1つのファイルで完結するため、配布や監査が容易です。
スクリプトには元のスキーマがどのようなチェックを行うかが直接コードとして書かれており、レビュー時の見通しもよくなります。

正規表現については、Pythonの`re.fullmatch`に相当する完全一致を、bashの`[[ ... =~ ^pattern$ ]]`で再現します。
bashの正規表現はPOSIX ERE準拠であり、Python固有の構文（例: `\d`、`\w`など）はそのままでは動作しないという制約はありますが、初期仕様としては実用的です。

## 検討したが採用しなかった案

### POSIX shスクリプトを生成する案

POSIX shで生成すれば`/bin/sh`があれば動かせますが、配列、`[[ ... =~ ... ]]`による正規表現、`shopt`によるオプション制御などが使えなくなり、`require`/`forbid`の`path_regex`や`count`をシンプルに表現できません。
今回は実装の単純さと出力の読みやすさを優先してbashを採用します。

### スキーマファイルを実行時に読み込むラッパースクリプトを生成する案

ラッパースクリプト方式（`yq`等で実行時にYAMLを読み込む）は実装が単純ですが、`yq`への依存が発生し、スタンドアロンという目的に反します。
この案ではYAMLパーサがランタイムに必要となるため、採用しません。

## 影響

`fs-schema`に新しいサブコマンド`compile`が追加されます。
既存コマンド（`validate`、`dump`、`validate-schema`、`validate-dump`、`show-schema`、`show-dump-schema`）の動作には影響しません。

bashの正規表現はPOSIX ERE準拠であるため、Python固有の正規表現構文を使うスキーマは生成されたスクリプトでは正しく動作しない可能性があります。
この点はドキュメントで明示し、必要に応じてユーザがスキーマを調整することを想定します。
