# ADR-0005: `path`の末尾`/`を不正とする

* ステータス: 承認済み

## 決定

スキーマ中の`path`プロパティでは、末尾が`/`で終わる文字列を不正入力として扱います。

これは`kind: directory`の場合も例外ではありません。
ディレクトリは`path: src`のように表現し、`path: src/`は許可しません。

将来的にも、末尾`/`を警告付きで受理したり、`kind: directory`のときだけ必須にしたりはしません。

## 理由

`path`はスキーマルートからの純粋な相対パスを表す単純なプロパティとして維持したいです。

ここで`src`と`src/`の差に意味を持たせると、`path`の文字列表現にエントリ種別の情報が入り込みます。
すると、現在`kind`が担っている責務と重複し、`path`と`kind`の両方を見ないと意味が確定しない設計になります。

特に、`kind: directory`のときだけ末尾`/`を要求すると、

* `path`が純粋なパスではなくなる
* `path`と`kind`の責務が二重化する
* `path_regex`の評価対象にもディレクトリ表現上の都合が混入する

という問題が生じます。

また、警告付きで`path: src/`を受理する案も検討しましたが、この場合は入力値の正規化とは別に警告収集と通知の仕組みが必要になります。
現状の実装は、スキーマ読み込み時に値を正規化し、不正値はエラーとして扱う単純な構成です。
末尾`/`だけを警告扱いにすると、この構成に対して複雑さが増えます。

そのため、`path`は末尾`/`なしの正規化済み相対パスのみを受け付ける方針とします。

## 影響

スキーマ利用者は、ディレクトリを含めて`path`に末尾`/`を書けなくなります。
エントリ種別の指定は引き続き`kind`で行います。

この方針により、`path`はファイル・ディレクトリ・シンボリックリンクのいずれについても同じ表現規則を持ちます。
`path_regex`も、エントリ種別とは独立した「ルート相対パス文字列」に対する条件として扱いやすくなります。

実装上は、スキーマバリデーションとJSON Schemaで末尾`/`を拒否する必要があります。
